① Job Submission Flow
User / Extension
1.1
Open Extension
JupyterLab sidebar loads the Puhti Run panel. Extension calls GET /containers to populate the environment dropdown.
1.3
Configure & Submit Job
User selects environment, uploads .ipynb notebook and optional requirements.txt, sets partition / CPUs / memory / time.
POST /run-notebook notebook: .ipynb requirements: optional partition: gpu | small | … container: pytorch3 time_hours: 2
1.8
Poll Status
Extension polls GET /status/{job_id} every 5 s. Shows live status chip: queued → running → done / failed.
1.10
View Results
Downloads stdout, stderr, output files from GET /results/{job_id}. Inline plots rendered in the extension.
FastAPI · Head Node
1.2
List Environments
SSH to Puhti — scans /scratch/…/envs/*/bin/python. Returns list of ready venv names.
SSH ControlMaster reuse cached 60 s
1.4
Validate Request
Authenticate JupyterHub token via x-jupyterhub-token header. Check per-user quota.
token valid ≤ MAX_CONCURRENT jobs 429 if over quota
1.5
Convert & Stage
nbconvert .ipynb → script.py, strip IPython magics. Create UUID job dir. rsync all files to Puhti scratch.
rsync -az job_dir/ → /scratch/…/runs/user/UUID/
1.6
Submit to Slurm
SSH sbatch command with env, resource flags, and job dir.
sbatch --partition=gpu --gres=gpu:v100:1 --export=ALL, VENV_PATH=.../envs/pytorch3, JOB_DIR=.../runs/user/UUID, USE_GPU=1 generic_run.sh
slurm_id stored in DB
1.9
Status Check
squeue -j slurm_id → running/done. On done: rsync results from Puhti → NFS. Update DB. Send email if failed (includes stderr tail).
Puhti · Slurm
1.7
Compute Node Runs Job
generic_run.sh executes on allocated node. Full isolated run inside user's venv.
module load python-data PYTHON=$VENV_PATH/bin/python # install job requirements $PIP install -r requirements.txt --target .packages/ # run PYTHONPATH=.packages \ HOME=$JOB_DIR \ $PYTHON script.py # stdout/stderr → files
GPU v100 via --nv each job: own UUID dir pip installs isolated
External Services
1.9b
Email Notification
SMTP (Gmail) sends completion or failure email to user. Failure emails include last 50 lines of stderr.
Job done ✓ Job failed ✗ + stderr
② Environment Request & Build Flow
User / Extension
2.1
Request Environment
User fills in a name and package list in the extension's "Request Container" tab.
POST /request-container-simple name: pytorch3 packages: torch torchvision matplotlib username: alice email: alice@uni.fi
2.3
Await Approval
Extension polls GET /my-container-requests/username. Shows status badge: pending → building → ready.
2.8
Environment Ready
User receives email. New env name appears in the job submission dropdown. Submit jobs using it immediately.
visible in dropdown
FastAPI · Head Node
2.2
Store Request
Validate name (lowercase + hyphens only), validate package list. Insert into container_requests table.
INSERT container_requests container=pytorch3 packages="torch torchvision …" status=pending username=alice
DB write 400 bad name / no packages
2.5
Submit Venv Build
On admin approval: generate build script, write to Puhti via SSH stdin, sbatch on small partition. Update DB with Slurm job ID.
_ssh_write(build-pytorch3.sh) sbatch build-pytorch3.sh → slurm_job_id=35457153 UPDATE container_requests SET status=building, build_job_id=35457153
2.7
Background Poller
Daemon thread runs every 60 s. Checks all building rows via sacct. On COMPLETED verifies venv python binary exists.
sacct -j 35457153 → State: COMPLETED test -f envs/pytorch3/bin/python → OK UPDATE status=ready → send email to alice
COMPLETED + python ✓ → ready FAILED/TIMEOUT → failed
Admin Panel
2.4
Review Request
Admin opens Environment Requests tab at /puhti/admin. Sees user, env name, requested packages, and status badge.
Approve Reject
2.6b
View Build Log
Admin clicks View Log at any time. API SSHes to Puhti and streams the Slurm output file content into a modal overlay.
GET /admin/container-log/42 → cat envs/build-pytorch3- 35457153.out
Puhti · Slurm
2.6
Build Venv
Small-partition Slurm job runs the build script. Finds Python ≥ 3.8 via module system. Creates isolated venv. Installs packages.
module load python-data # → Python 3.11 VENV=/scratch/.../envs/pytorch3 python3 -m venv $VENV # torch: cu121 wheels $VENV/bin/pip install \ --index-url …/whl/cu121 \ torch torchvision # other packages $VENV/bin/pip install matplotlib echo VENV_BUILD_SUCCESS
TMPDIR → scratch (no /tmp overflow) ~5 min total
External Services
2.5b
GitHub Record
On approval, API commits envs/pytorch3.txt to GitHub repo — tracks who requested it, what packages, when approved, and the Slurm build job ID.
envs/pytorch3.txt: name: pytorch3 requested_by: alice approved_at: 2026-07-15 slurm_build_job: 35457153 packages: - torch - torchvision - matplotlib
2.8b
Ready Email
SMTP sends email to user once poller confirms venv is ready. Includes env name and instructions to select it from the dropdown.
ready to use
③ Checks & Guards Reference
Authentication
Every job submission validates x-jupyterhub-token against JupyterHub API. Unauthenticated requests receive 401.
Per-user Quota
Max concurrent active jobs per user enforced before sbatch. Excess requests receive 429 Too Many Jobs.
Per-user 100 GB Disk Quota
Total output size checked across all jobs for user. Exceeding 100 GB blocks new submissions with 507 Insufficient Storage.
Admin Auth
GitHub OAuth via HMAC-signed session cookie. Only users in ADMIN_ALLOWED_USERS env var can access the admin panel.
CORS Lockdown
Only hbv.we3data.com and diwa-data-lab-vre.rahtiapp.fi allowed. Controlled via CORS_ALLOWED_ORIGINS env var.
Partition Time Limits
Each partition has a max wall-time cap enforced by the API before sbatch. Users cannot request more than the partition limit.
Env Name Validation
Container names must match ^[a-z0-9-]+$. Prevents path traversal and shell injection in Slurm export args.
Rate Limit & Logging
All key events logged: job_submitted, status_change, sbatch_failed, rate_limit_hit, venv_ready. Rotating log 10 MB × 5 files.
